# Markdown issue #495 性能修复复盘

## 结论

最终方案没有延迟或拆分 Markdown AST：知乎 HTML 会一次性转换为完整 `Document`，issue #495 样本稳定得到 406 个节点，其中 148 个公式节点。真正延迟的是屏外顶层块的 Compose 组合、测量和布局。

`deferOffscreenBlocks` 在内置 renderer 和 Zhihu++ 两层入口都默认启用。首帧只真实布局视口以及上下各半屏的预取范围；其他顶层块只生成高度占位。块滚出预取范围后会重新变成占位，已经真实布局过的块会复用实测高度，不再退回粗略估算。

最终代码基于上游实际版本 `1.2.9`。应用替换前使用的是分叉构件 `0.0.1-alpha.11`，这个构件版本不是上游 base version。实验工作树中的后续 parser、布局引擎和自定义选择实现没有整体带入；特别是仍有选择问题的自定义选择实现被排除，继续使用基线的原生 Compose `SelectionContainer`。

## 测试样本与方法

- 样本：`app/src/androidTest/assets/issue-495-answer.html`
- 大小：39,620 bytes、36,460 个字符
- AST：406 个节点、148 个公式节点，其中 58 个独立公式块、90 个行内公式
- Android 环境：本地 Android 31 ARM64 AVD，lite debug APK
- 首帧口径：同一测试进程先用小公式页面预热 Compose 和 ArticleScreen，再用 2 次完整 issue 样本预热密集公式路径，随后连续 5 次计时真实 `ArticleScreen` 并取中位数；每轮追加一个不改变渲染结果的唯一 HTML comment，强制 `remember(html)` 失效，避免复用上一轮 AST 或块测量状态。每次计时覆盖 HTML 转 AST、Compose 组合、测量、布局，直到根节点完成一次截图，不包含测试进程和 Activity 启动。报告同时记录完整样本的预热耗时，避免只呈现热路径数字
- AST 口径：同进程 20 次预热，随后 50 次计时，避免把 JVM 启动和冷类加载算入解析阶段

公式归因测试只用于定位布局成本。测试通过把同一份 HTML 中的公式标签临时替换成廉价文本来比较时间，不是产品实现，也不代表最终会删除或省略公式。

## 分阶段结果

| 方案或阶段 | 结果 | 结论 |
| --- | ---: | --- |
| 完整 HTML 转完整 AST，50 次计时中位数 | 2.42 ms | AST 构建不是卡顿点 |
| 完整 HTML 转完整 AST，P90 | 3.57 ms | 不应破坏 AST 完整性来优化这一阶段 |
| 全量 eager 布局全部公式 | 6,310 ms | 真实卡顿主要位于 Compose 布局和公式渲染 |
| 诊断性替换 58 个独立公式块，保留 90 个行内公式 | 3,180 ms | 独立公式块贡献约 3,130 ms |
| 诊断性替换全部 148 个公式 | 2,409 ms | 非公式正文全量布局本身仍超过 2 秒 |
| 只解析前三个 HTML 块的旧方案 | 约 1,268 ms | 数字快，但 AST 不完整，方案无效 |
| 最终方案的 2 次完整样本预热 | 1,104 ms、723 ms | 记录完整公式路径进入稳定状态的过程 |
| 最终完整 AST + 全顶层块视口懒布局，5 次中位数 | 406 ms | 样本为 474、406、414、388、361 ms；较 6,310 ms 降低约 93.6% |

独立公式块约占完整首帧时间的 49.6%，占公式相关额外成本的约 80.2%。因此“只延迟独立公式块”确实能显著改善，但仍会留下约 3.18 秒首帧，并不能达到当前结果。最终选择对所有屏外顶层块统一延迟布局，公式块自然包含在其中。

## 滚动范围预测

高度不是一个固定常量，而是逐块估算：

- 段落按可用宽度、中文和 ASCII 字符宽度、显式换行、行内代码及行内公式估算行数；
- 公式块按公式行数和分式、根号、求和、积分等高结构增加高度；
- 代码块按行数和内边距估算；
- 图片优先使用宽高比，缺少尺寸时使用保守默认比例；
- 列表、引用、脚注等容器递归累加子块高度和块间距。

在最终 API 31 AVD 复测中：

- 首帧预测最大滚动范围：38,301 px
- 全文滚动并实际布局后的最大范围：37,955 px
- 预测高出 346 px，即 0.91%

instrumented test 还保留了 25% 的跨环境容差断言，防止以后高度估算退化到明显误导进度条的程度。块首次真实布局后会记录实际高度；它再次离开预取范围时使用实测高度，因此预测会随滚动逐步校准，而不是反复回到字符粗估。

## 最终实现

1. HTML 只调用一次 `htmlToMdAst`，完整 AST 从首帧开始就可用于脚注、图片预览、标题和其他全局结构。
2. `SelectableColumn` 和 `StaticColumn` 的顶层块默认进入视口懒布局；已有 `LazyColumn` 模式继续使用自身的 item 懒布局。
3. 每个屏外顶层块只保留一个估高占位。进入视口上下半屏预取范围后才调用真实 `BlockRenderer`。
4. 块离开预取范围后恢复占位；若已经测量过，则占位使用实测高度，减少滚动位置和进度范围跳动。
5. 脚注定义或引用位于屏外时，导航状态只临时物化对应顶层块，注册 `BringIntoViewRequester` 后完成跳转，再清除强制物化标记。
6. 脚注行内实现仍来自上游 PR #15，并保留评论提出的像素宽度 `+1px` 修复，避免 Android 像素到 sp 舍入裁掉末尾。

## 尝试过但拒绝的方案

### 整体采用实验工作树或后续上游实现

最初错误地把 issue 实验工作树当成应该整体内置的新版来源。这会带入后续 parser、布局引擎和自定义选择等大量无关改动，也会继承仍未解决的选择回归。最终回到应用真实依赖边界：以上游 `1.2.9` 为 base，只保留应用原有必要兼容和经过本项目真实测试的修复。

### 拆分 HTML、逐段生成 AST

这个方案首帧约 1.27 秒，但代价是首帧 AST 只含前缀。脚注表、预览图片、跨块结构和全局后处理都不完整，滚动范围也只能按已加载前缀计算。AST 阶段实测中位数只有 2.42 ms，因此该方案优化了错误阶段并破坏了数据契约，已完全删除。

### 只处理公式块

诊断证明独立公式块是最大单项成本，但只处理它们仍剩约 3.18 秒。最终没有新增“删公式”或“只加载公式”的特殊数据流，而是让所有屏外顶层块遵循同一视口规则。这样既覆盖公式，也覆盖图片、代码、长段落和复杂容器。

### 只保留已访问块、不回收

这个中间版本能把首帧降到约 1.2 秒，但滚动过的块会永久保留真实布局，长文浏览越深，Compose 树越接近全量。最终改为离开预取范围就回收，并用实测高度替代初始估高。

## 本次错误与根本原因

1. **基线识别错误。** 曾把开发工作树版本和应用实际依赖版本混为一谈，也曾把 `0.0.1-alpha.*` 写成上游 base version。根因是没有先把“上游发布版”“用户分叉构件”和“实验工作树”三个概念分开。
2. **来源目录判断错误。** 用户指定的本地 issue 工作树应当作为候选实现证据，而不是自动替换成远端最新版；后续又反过来把实验工作树整体当基线。正确做法是先固定应用真实基线，再逐项审查候选修复。
3. **benchmark 绕开真实瓶颈。** 早期测试通过预构造结果或缩短内容，数字没有覆盖真实 HTML 转换、Compose 测量、布局和绘制，无法证明用户看到的卡顿已修复。
4. **没有先分阶段计时。** 在测出 AST 只有约 2.42 ms 之前就假设 AST 解析很慢，导致设计了不完整 AST 的增量方案。性能改动前必须先测输入转换、结构构建、测量、布局和绘制各自成本。
5. **把完整性当作可交换成本。** 拆 HTML 前缀虽然快，却破坏脚注、图片预览和全局结构。数据结构完整性不能为了一个未经归因的性能目标被牺牲。
6. **进度范围只覆盖已加载前缀。** 旧分页让 `scrollState.maxValue` 只代表当前批次，进度条会随着加载不断改变含义。最终用所有顶层块的占位高度从首帧预测全文范围。
7. **公式诊断措辞错误。** “替换公式做归因测试”曾被说成“去除公式优化”，混淆了实验手段和产品方案。诊断变体不应被描述为最终行为。
8. **默认值不符合性能功能定位。** 作为长文默认渲染策略，`deferOffscreenBlocks` 必须默认 `true`，不能要求每个调用点显式打开。
9. **使用了被裁剪的视口坐标。** 初版用 `boundsInWindow()` 判断块是否接近视口，屏外坐标被裁剪到边界，导致全部块误物化。改用未裁剪的 `positionInWindow()` 与实测高度后，屏外语义节点才真正消失。
10. **没有同时设计懒布局下的脚注导航。** 屏外脚注定义和返回引用都可能尚未注册 requester。最终增加按目标临时物化，分别覆盖跳到定义和返回引用两条路径。

## 保留的行为与已知取舍

- 完整 AST、脚注结构、图片预览列表和 footer 从一开始就是完整的。
- 原生 Compose `SelectionContainer` 保留，没有采用实验工作树的自定义选择实现。
- 屏外文本没有真实布局，因此只有物化范围内的文本参与当前 Compose 选择树；跨大范围选择若需要与全量 eager 布局完全相同的行为，需要另行设计，不能复用已知有 issue 的上游自定义选择方案。
- 公式等高成本会转移到滚动接近对应位置时发生。实现使用半屏预取降低突然卡顿，但公式极密集时不可能同时做到零首帧成本和零滚动成本。
- 未提供宽高的远程图片只能先用默认比例估高；图片实际显示后会用测得高度校准。

## 验证项目

- 完整 AST 测试：固定 406 个节点、148 个公式节点。
- 真实首帧测试：覆盖 HTML 转 AST、ArticleScreen 组合、layout 和 draw；完整样本预热为 1,104、723 ms，随后 5 次测量中位数 406 ms，门槛 5,000 ms。
- 懒布局测试：首屏确认后段文本不存在；滚动后后段文本出现，同时首段退出语义树。
- 进度预测测试：初始范围必须在全文实测范围的 75% 到 125% 之间；当前误差 0.91%。
- 脚注测试：外层 `ScrollState` 下，从首屏引用跳到屏外定义，再返回已回收的首屏引用。
- 构建与格式化：按项目要求先执行 `assembleLiteDebug`，再执行 `ktlintFormat`，格式化后重新构建和运行定向测试。
